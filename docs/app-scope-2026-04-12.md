# App Scope - 2026-04-12

## Objetivo del documento

Este documento resume el alcance actual de Serenity para demos y conversaciones de producto.

La intencion es dejar claro:

- que ya existe en la app,
- que existe como demo operativa,
- que existe solo como direccion de producto,
- y que todavia no forma parte del alcance actual.

## Estado general actual

Serenity ya funciona como una demo operativa multi-actor para el flujo principal de:

1. creacion de demanda,
2. cobertura operativa,
3. ejecucion basica,
4. revision,
5. trazabilidad.

Hoy la app ya permite mostrar una historia coherente entre:

- `Center manager`
- `Provider coordinator`
- `Provider reviewer`
- `Carer`

La cobertura funcional es desigual entre actores:

- `provider` es la superficie mas madura
- `center` ya es util para demos y seguimiento
- `carer` ya tiene Fase 1 operativa y una base clara de readiness
- cierre operativo ya es visible desde UI
- export package inicial ya existe
- sync externo ya existe con ciclo visible simplificado, runner programable por tiempo, chequeo remoto mock y retries
- `xero_custom_connection` ya existe como primer partner de integracion foundation
- cierre financiero completo y backoffice avanzado siguen fuera del alcance actual

## Alcance actual por actor

### 1. Provider coordinator

Existe hoy:

- dashboard operativo
- lista de ordenes
- vista de `closing`
- periodos de cierre
- settlements por visita aprobada
- visibilidad de visitas excluidas del settlement con motivo y siguiente paso
- visibilidad de approved minutes, billable y payable
- gastos y kilometraje basicos por visita aprobada
- estados `open`, `locked` y `exported`
- descarga de export package en `json` y `csv`
- trazabilidad basica de exportacion
- export jobs explicitos
- estado visible de sincronizacion por job
- referencia externa mock del sistema receptor
- procesamiento de entrega por job
- ejecucion por lote de jobs encolados
- chequeo remoto de jobs `sent`
- historial de intentos por job
- endpoint interno seguro para disparar el runner sin sesion de usuario
- acuse o rechazo externo simulado
- retry de jobs fallidos
- `xero_custom_connection` con modos `sandbox`, `auth_only` y `direct_post`
- creacion de ordenes
- edicion de ordenes
- agregado de visitas
- asignacion de carers elegibles
- razones visibles de por que un carer no es elegible
- reemplazo manual de visitas
- escalamiento operativo desde detalle de orden
- estados visibles de cobertura por orden y visita
- cambio de estado de visitas
- visibilidad de checklist, evidencia, incidentes y auditoria

No existe hoy:

- aprobacion o rechazo de visitas
- creacion de incidentes desde la vista provider
- payroll nativo
- sincronizacion bidireccional con sistema externo
- confirmacion remota real del sistema externo
- el endpoint final de handoff real hacia Xero o middleware

### 2. Provider reviewer

Existe hoy:

- todo lo visible para provider
- aprobacion o rechazo de visitas en `under_review`
- actualizacion de audit trail al revisar

No existe hoy:

- flujo avanzado de observaciones multi-etapa
- cola de revision especializada con filtros o productividad

### 3. Center manager

Existe hoy:

- dashboard de demanda y cumplimiento
- lista de ordenes del centro autenticado
- creacion de solicitudes de servicio
- detalle de orden
- visibilidad de cobertura, incidentes, evidencia y review
- aislamiento por centro

No existe hoy:

- asignacion de carers
- cambio de estados de visitas
- aprobacion o rechazo
- control fino sobre operacion provider

### 4. Carer

Existe hoy:

- agenda de visitas asignadas
- detalle operativo de la visita
- visualizacion de instrucciones y skills requeridas
- checklist basica editable
- evidencia basica por referencia o URL
- reporte simple de incidencias
- visualizacion de skills verificadas
- readiness del perfil (`ready`, `attention_needed`, `restricted`)
- alertas visibles de credenciales vencidas o proximas a vencer
- limites operativos visibles en UI
- disponibilidad editable por nota
- bloques de disponibilidad
- credenciales con estado y vencimiento visible

No existe hoy:

- carga real de archivos
- experiencia mobile-first refinada
- historial economico y gastos
- alertas automaticas fuera de la UI

## Alcance demo actual

En demos, hoy Serenity puede presentarse como:

**una plataforma de coordinacion y ejecucion para homecare con trazabilidad multi-actor.**

La narrativa demo actual mas fuerte es:

1. el centro crea una necesidad,
2. la prestadora recibe y organiza cobertura,
3. el cuidador ejecuta y registra evidencia,
4. la prestadora revisa y aprueba,
5. el sistema conserva auditoria.

## Lo que conviene decir explicitamente en demos

- Esto es una demo operativa, no un producto financiero completo.
- El modulo `closing` ya muestra consolidacion operativa, pero no ejecuta pagos.
- El modulo `carer` ya tiene ejecucion basica, pero no backoffice completo.
- El perfil del carer ya permite editar disponibilidad y gestionar credenciales de forma basica.
- No es un EMR clinico.
- No intenta cubrir toda la contabilidad ni todos los flujos clinicos.

## Fuera de alcance actual

Hoy no forma parte del alcance actual:

- facturacion asistida
- liquidacion del cuidador
- reglas economicas complejas
- experiencia avanzada de reemplazos
- validacion end-to-end local del nuevo seed operativo
- cargador real de archivos
- alertas de documentos vencidos
- app familiar o de paciente
- EMR clinico

## Que falta despues de este bloque

Para pasar de esta base de sync a una integracion util de verdad, Serenity todavia necesita:

- ejecucion realmente desacoplada por worker o cron
- confirmacion remota real del sistema receptor
- politicas de retry mas robustas
- conectores por plataforma externa
- observabilidad tecnica de integracion

## Estado de Fase 2 del carer

La base de Fase 2 ya esta iniciada.

Hoy Serenity ya cubre:

- disponibilidad editable por nota
- bloques de disponibilidad
- credenciales con estado
- vencimiento visible en UI
- readiness visible
- alertas in-app
- explicacion de limites operativos

Lo siguiente para profundizar esta fase seria:

- cierre de validacion end-to-end con base levantada
- pulido de UX de alertas y restricciones
- experiencia mas refinada de perfil profesional
